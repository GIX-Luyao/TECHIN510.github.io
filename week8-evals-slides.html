Three Different Questions

Tests, Evals, Observability

Layer Answers Verdict shape When it runs
Tests Did the code run? Pass / fail On every commit
Evals Was the output good? Score, distribution On prompt / model change
Observability Is it still good in production? Trends, anomalies, drift Continuously

In practice people use test, validation, and eval interchangeably

Wk Pattern What you did

Why Tests Stop Working

Failures Your Asserts Won't Catch

1
The output is different every run draft-cover-letter never produces the same text twice. assert draft == expected is meaningless — there is no fixed expected value to compare against.
2
Structure is right, content is wrong match-resume returns {score: 87, reasons: [...]}. Schema check passes. But "reasons" cites a TensorFlow project the candidate never did. Hallucinated content, valid shape.
3
User constraints quietly violated USER.md says "never use the word synergy." Cover letter uses synergy three times. No assert catches this without rewriting half the eval.

Tests stop where determinism stops. To score these you first need to see them.

Free-Form Output

Assertion Fails

Generated

Dear Anthropic team,

I'm writing to apply for the Research Engineer role. My work on three production ML systems makes me a strong fit for your team.

Also Valid

Dear Anthropic hiring committee,

I'd like to be considered for the Research Engineer position. Having built three production-scale ML systems, I believe I'd be valuable to your team. 

assert generated == expected   → False

There is no single expected for free-form output. Score against criteria, not against a reference.

07 / 18
SCORING METHOD

Schema Scoring

Deterministic checks on structured output. Pass / fail. No judgment, no judge. 

output = run_skill("match-resume", goal)

assert output["company"]
assert output["role"]
assert isinstance(output["score"], int)
assert 0 <= output["score"] <= 100

When to use: structured outputs you control the shape of. Cheap. Brittle to format drift.

08 / 18
SCORING METHOD

Rubric Scoring

A rubric is your criteria. Score each output independently; no reference needed.

  • Relevance — addresses the actual user goal
  • Factuality — no invented experience
  • Format — matches your SOUL.md voice

When to use: any free-form output. Cost: human or judge time per case.

09 / 18
SCORING METHOD

Pairwise Comparison

Ask "is A better than B?" instead of "how good is A?" — a controlled experiment for your evals.

  • Control: golden set, judge, rubric — held fixed
  • Treatment: change one thing (model, prompt, parameter)
  • Outcome: A wins / B wins / tie → win-rate

Humans (and judges) calibrate relative judgments better than absolute scores. Change two things at once and you can't tell which one moved the number.

AUTOMATION

LLM-as-Judge

# judge.py — the entire pattern in 12 lines
def judge(user_goal, agent_output, rubric):
    prompt = f"""
You are evaluating an agent's response.

USER GOAL: {user_goal}
AGENT OUTPUT: {agent_output}

Score 0–3 on each dimension. Return JSON only.

RUBRIC:
{rubric}
"""
    response = anthropic.messages.create(
        model="claude-sonnet-4-6",   # different family from prod
        max_tokens=500,
        messages=[{"role": "user", "content": prompt}],
    )
    return json.loads(response.content[0].text)

When to use: rubric scoring at scale. Catches what humans miss — and lies in predictable ways (next slide).

11 / 18
When The Judge Lies

Three Failure Modes

1
Position bias In A/B comparisons, the judge prefers whichever option you list first. Fix: randomize order, run both ways, average.
2
Verbosity bias Longer outputs score higher even when shorter is better. Fix: normalize for length, or score concision explicitly in the rubric.
3
Self-preference Claude judges prefer Claude outputs over GPT outputs, even when blinded. Fix: judge with a different model family than production.

Always spot-check. Pick three judge scores per run and read them yourself. The judge is a tool, not an oracle.

DATA

Golden Set

Golden set = hand-picked example inputs with expected behavior, frozen as your control. Same set every run. 

input: "Apply to the Anthropic research engineer role"
expected:
  word_count: < 450        ← checked by schema (P1)
  references:                ← checked by faithfulness (P5)
    - Anthropic's research
    - candidate's projects
  banned_words: [synergy]    ← checked by schema (P1)

10–15 of these = your golden set. Commit to eval/golden.jsonl. Each pattern that needs a reference reaches in for its slice.

WORKFLOW

Regression + Ship Memo

The five-step loop
  1. Freeze the golden set. No edits.
  2. Run on both old and new versions.
  3. Diff aggregate, per-case, latency, cost.
  4. Spot-check 3 regressions by hand.
  5. Memo the decision in 3 sentences.

Trigger: model bump, prompt change, skill rewrite.

Ship memo template
# Sonnet 4.6 → 4.7 (rubric 0–3)

| Metric        | 4.6  | 4.7  | Δ     |
|---------------|------|------|--------|
| Relevance     | 2.4  | 2.7  | +0.3   |
| Factuality    | 2.6  | 2.5  | −0.1   |
| Format        | 2.5  | 2.6  | +0.1   |
| Aggregate     | 2.50 | 2.60 | +0.10  |

Regressions: Case 07, Case 12
Recommendation: Ship. Add banned-words
post-check before finalize.

When to use: any time something upstream of your agent changes.

Before You Trust The Deltas

Statistical Rigor

1
LLMs are stochastic — run each case multiple times Temperature 0 is not always reproducible. Run N=3 per case at minimum. Report mean and variance, not a single score.
2
15 cases × 3 runs = 45 data points Anything less is anecdote. A Δ0.1 on n=15 is noise; on n=150 it is signal. Confidence intervals decide.
3
Measure judge–human agreement Spot-check 20% of judge scores by hand. Compute Cohen's kappa. Below 0.6 means your rubric is ambiguous — fix the rubric, not the agent.
4
Bootstrap confidence intervals Five lines of scipy give you a 95% CI on any score delta. Add it to your eval runner once; never argue noise-vs-signal again.

Key Takeaways

Key Takeaways

1
Tests tell you it ran. Evals tell you it was good. Two different questions, two different tools. Confusing them is the most common AI-product failure.
2
Three things to build: scoring methods, data, pipeline. Scoring methods (schema, rubric, pairwise, judge) tell you good vs bad. Data (golden set) is what you score against. Pipeline (trace, regression) is how you run it every time something changes.
3
Trace first, eval second. One row per step answers all three questions at once: where did it break, what's slow, what cost too much. Everything else sits on top of the trace.
4
The ship memo is the deliverable. Same rubric dimensions, old vs new, regressions called out, recommendation in one line. Bring this every time you change a model or a prompt.
